# CONTEXT для AI-агента

> Этот файл — краткая выжимка о проекте для AI-помощника (Claude Code и т.п.). Прочитай его в начале сессии, чтобы понять, где мы находимся и куда движемся. Файл обновляется по мере работы.

## О проекте

**ЯсноТекст** — учебный проект по дисциплине «Основы проектной деятельности». Десктопное приложение на C# (WPF, .NET 10), которое превращает «недоступные» цифровые документы (PDF, DOCX, сканы) в удобную среду для людей с нарушениями зрения и дислексией.

**Целевая аудитория:** школьники и студенты с дислексией (ядро), слабовидящие пользователи, незрячие (через TTS).

**Срок:** к концу учебного семестра.

**Репозиторий:** https://github.com/Serouke/yasnotext

## Технологический стек

- **.NET 10** + **WPF** (C# + XAML)
- **xUnit** для юнит-тестов
- **PdfPig** (`UglyToad.PdfPig`, prerelease 1.7.0-custom-5) для парсинга PDF
- **DocumentFormat.OpenXml** для DOCX (планируется в коммите 6)
- **Tesseract.NET** для OCR (планируется в коммите 7)
- **System.Speech.Synthesis** для TTS (планируется в коммите 10)

## Структура решения

Три проекта в одном solution:

```
src/
├── YasnoText.Core/         ← бизнес-логика, не зависит от WPF
│   ├── Profiles/           ← модели профилей чтения, ProfileManager
│   └── DocumentReaders/    ← IDocumentReader, PdfTextReader, DocumentResult
├── YasnoText.UI/           ← WPF-приложение
│   ├── Themes/             ← ResourceDictionary темы + ThemeManager + WpfThemeApplier
│   ├── ViewModels/         ← MVVM (ViewModelBase, RelayCommand, MainViewModel, ProfileItemViewModel)
│   ├── App.xaml(.cs)
│   └── MainWindow.xaml(.cs)
└── YasnoText.Tests/        ← юнит-тесты на xUnit
```

Дополнительно:
- `test-files/` (в корне репо) — PDF для юнит-тестов
- `docs/` — документация (architecture.md, CONTEXT.md, инструкции по коммитам)

## Архитектурные решения и почему так

**Core ничего не знает о WPF.** Все классы в `YasnoText.Core` — обычные POCO и не зависят ни от чего, кроме .NET Standard. Это даёт два плюса: легко тестировать без WPF-контекста, и в будущем можно сделать другой UI (web, mobile) без переписывания логики.

**Связь Core → UI через интерфейсы.** `IThemeApplier` объявлен в Core, реализован в UI как `WpfThemeApplier`. `MainViewModel` принимает интерфейс через конструктор — классическая инверсия зависимостей.

**MVVM с ItemsControl.** Карточки профилей рендерятся не вручную (через 3 захардкоженных Border), а через `ItemsControl` с `DataTemplate`. Это значит: добавишь четвёртый профиль — UI автоматически отрисует четвёртую карточку.

**Темы через ResourceDictionary.** Три файла (`StandardTheme.xaml`, `LowVisionTheme.xaml`, `DyslexiaTheme.xaml`) с одинаковыми ключами и разными значениями. Переключение — через `Application.Current.Resources.MergedDictionaries.Clear() + Add(...)`. Все элементы в `MainWindow.xaml` используют **`DynamicResource`** (не `StaticResource`), чтобы реагировать на смену темы в рантайме.

**Подсветка активной карточки — через `DataTrigger`.** Изначально я попробовал через `IValueConverter`, но он отрабатывал до полной загрузки темы и давал случайные цвета. `DataTrigger` на свойстве `IsActive` в `ProfileItemViewModel` — нативный WPF-способ, работает правильно.

**Для отображения текста документа — `TextBox` в режиме read-only.** Не `FlowDocumentScrollViewer` — он плохо биндится к динамическому тексту. `TextBox` отлично работает с биндингом и выглядит достаточно для MVP.

## Известные особенности и грабли

**1. PdfPig prerelease.** Стабильной версии нет, ставится с флагом `--prerelease`. Версия `1.7.0-custom-5` стабильна для нашего MVP.

**2. LF vs CRLF.** При коммите Git ругается `LF will be replaced by CRLF`. Это нормально для Windows, никак не влияет на работу. Можно подавить через `git config --global core.autocrlf true`, но не обязательно.

**3. `.gitignore` и тестовые PDF.** В `.gitignore` есть строка `test-files/`, которая блокирует коммит тестовых PDF целиком. Чтобы тесты работали при клонировании репо, нужно эту строку удалить и закоммитить четыре PDF из `test-files/`. Размер каждого PDF — десятки килобайт, репо не раздуется.

**4. Шрифт OpenDyslexic — заглушка.** В `DyslexiaTheme.xaml` пока стоит `Georgia` вместо реального OpenDyslexic. Реальный шрифт нужно скачать (OFL-лицензия, бесплатно), положить в `src/YasnoText.UI/Resources/Fonts/`, прописать через pack-URI. Это коммит 11.

**5. Системный заголовок окна.** На Windows 11 светлая плашка `ЯсноТекст` сверху окна не меняется при смене темы — это компонент ОС, без хаков не настраивается. На защите этот нюанс упоминать не нужно.

**6. PdfPig склеивает абзацы без пробелов.** На некоторых PDF (особенно сгенерированных reportlab) `page.Text` возвращает текст со склейками вроде «обучениянейробиологического». В коммите 7 (OCR) добавим `TextProcessing` с постобработкой, которая исправит и эти склейки, и переносы.

**7. ComboBox со шрифтами.** В боковой панели сейчас захардкожен `Foreground="Black"`. Стилизация ComboBox под все темы — отдельная большая работа, для MVP допустимо.

**8. Неработающие элементы UI.** Слайдеры «Межбуквенный интервал» и «Высота строки», кнопки `A−`/`A+`, кнопки `Сохранить` и `Озвучить`, ComboBox шрифтов — пока **визуальные заглушки**, не привязаны к логике. Это коммит 8 (настройки) и коммит 10 (озвучка).

## Стиль работы

**Маленькие коммиты с понятными сообщениями.** Conventional Commits: `feat:`, `fix:`, `perf:`, `chore:`, `docs:`, `test:`, `refactor:`. Один коммит = одна логическая фича. Это не только хорошая практика, но и сильный аргумент на защите — «вот история разработки».

**Тесты сразу с фичей.** Каждый коммит с новой бизнес-логикой добавляет 3-10 юнит-тестов. К концу проекта целимся в 30+ тестов.

**Пользователь работает в VS Code на Windows.** Терминал — PowerShell. Команды пишутся в формате, который сразу копируется в PowerShell без модификаций.

**Архивы вместо отдельных файлов.** Когда AI-помощник готовит код для пользователя, удобно делать один ZIP-архив со всеми файлами и инструкцией, что куда копировать. Это удобнее, чем гонять файлы по одному.

## Совет следующему агенту

Когда подключаешься, прежде чем что-то делать:

1. Запусти `git log --oneline` — увидишь, что уже сделано.
2. Запусти `dotnet build` — убедись, что проект собирается.
3. Запусти `dotnet test` — должно быть 25/25 (или больше, если уже сделан коммит 6).
4. Спроси пользователя, на каком коммите он остановился и что хочет делать дальше.

Не предлагай переписать архитектуру «по красоте» — она уже выстроена осознанно, и переделка в середине проекта только запутает. Двигаемся маленькими шагами, добавляем фичи слой за слоем.
